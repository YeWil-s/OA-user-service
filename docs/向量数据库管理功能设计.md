# 向量数据库管理功能设计文档

> 版本: v1.0 | 日期: 2026-07-23 | 状态: 待开发

---

## 一、背景与目标

### 1.1 现状

当前 OA 系统使用 **Redis Stack (RediSearch)** 作为向量数据库，已打通知识文档的自动向量化（SiliconFlow BAAI/bge-large-zh-v1.5，1024维）、存储、KNN检索和 RAG 问答。但存在以下短板：

| 短板 | 说明 |
|------|------|
| 无公司维度 | 知识文档没有公司归属，无法支持多公司/多租户场景 |
| 仅有角色控制 | 访问控制只到角色级别（ROLE_ADMIN/HR/LEADER/EMPLOYEE），没有利用已有的岗位（sys_position）和部门树 |
| 无管理面板 | 没有向量索引统计、健康检查、批量操作、导入导出等管理工具 |
| 无审计追踪 | 知识库的增删改没有操作日志 |
| 无质量保障 | 入库前不去重，无相似度检测，无向量覆盖率监控 |

### 1.2 目标

- **精细化权限**：知识可标注所属公司，可指定哪些岗位/部门/角色的人能访问
- **可视化管理**：提供向量库仪表盘，支持索引监控、知识检索、批量操作、导入导出
- **质量保障**：入库去重检测、向量覆盖率监控、操作审计日志
- **向后兼容**：现有 API 和数据结构尽量兼容，新增字段设默认值

## 二、数据库设计

### 2.1 公司/组织归属：复用 `sys_dept` 表

> **不需要新建公司表。** 现有 `sys_dept` 表已支持部门树结构（`parent_id` 字段），其中 `parent_id=0` 的记录即为公司/一级组织。知识文档通过 `dept_id` 字段关联 `sys_dept.id`，即可实现公司维度的知识归属。
>
> 举例：`sys_dept` 中 `id=1, parent_id=0, dept_name='总公司'` → 这就是一个公司节点。

后端提供公司列表接口时，直接查 `sys_dept` 中 `parent_id=0 AND status=1` 的记录即可，无需额外建表。

### 2.2 改造表：`ai_knowledge_doc` 新增字段

```sql
ALTER TABLE ai_knowledge_doc
    ADD COLUMN dept_id          BIGINT       DEFAULT NULL   COMMENT '所属组织ID(关联sys_dept,parent_id=0即为公司)',
    ADD COLUMN access_positions VARCHAR(500) DEFAULT NULL  COMMENT '可访问岗位ID列表(JSON数组,如[1,3,5])',
    ADD COLUMN access_depts     VARCHAR(500) DEFAULT NULL  COMMENT '可访问部门ID列表(JSON数组)',
    ADD COLUMN access_mode      TINYINT      NOT NULL DEFAULT 0 COMMENT '权限模式: 0=任一满足, 1=全部满足, 2=角色+岗位同时满足',
    ADD COLUMN version          INT          NOT NULL DEFAULT 1 COMMENT '版本号(每次修改+1,用于追踪变更)',
    ADD COLUMN vector_status    TINYINT      NOT NULL DEFAULT 0 COMMENT '向量状态: 0=未向量化, 1=已向量化, 2=向量化失败',
    ADD COLUMN vector_error     VARCHAR(500) DEFAULT NULL   COMMENT '向量化失败原因',
    ADD INDEX idx_dept (dept_id),
    ADD INDEX idx_vector_status (vector_status);
```

字段说明：
- `dept_id`：所属组织（关联 `sys_dept.id`），通常指向 `parent_id=0` 的公司级节点
- `access_positions`：可访问的岗位ID列表，如 `[1,3,5]`，NULL 表示不限制岗位
- `access_depts`：可访问的部门ID列表，NULL 表示不限制部门
- `access_mode`：当同时设置了角色、岗位、部门时的判定逻辑
- `vector_status`：追踪每条知识的向量化状态，支持"未向量化"和"失败"的筛选和重试
- `version`：每次修改递增，用于对比 Redis 中的版本是否一致

### 2.3 新增表：知识操作日志 `ai_knowledge_log`

```sql
CREATE TABLE ai_knowledge_log (
    id           BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键',
    doc_id       BIGINT       NOT NULL             COMMENT '知识文档ID',
    operation    VARCHAR(20)  NOT NULL             COMMENT '操作类型: CREATE/UPDATE/DELETE/REINDEX/EXPORT',
    operator_id  BIGINT       NOT NULL             COMMENT '操作人ID',
    operator_name VARCHAR(50) DEFAULT NULL         COMMENT '操作人姓名(冗余,防用户删除)',
    detail       VARCHAR(1000) DEFAULT NULL        COMMENT '变更详情(JSON: 记录变更前后的关键字段)',
    ip_address   VARCHAR(50)  DEFAULT NULL         COMMENT '操作IP',
    create_time  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    KEY idx_doc_id (doc_id),
    KEY idx_operator (operator_id),
    KEY idx_create_time (create_time)
) ENGINE=InnoDB COMMENT='知识库操作日志';
```

### 2.4 新增表：向量索引快照 `ai_vector_snapshot`

> 定期记录向量库状态，用于监控趋势

```sql
CREATE TABLE ai_vector_snapshot (
    id            BIGINT   NOT NULL AUTO_INCREMENT,
    total_docs    INT      NOT NULL DEFAULT 0 COMMENT 'MySQL中启用状态的知识总数',
    indexed_docs  INT      NOT NULL DEFAULT 0 COMMENT 'Redis中已索引的向量数',
    failed_docs   INT      NOT NULL DEFAULT 0 COMMENT '向量化失败数',
    index_memory  BIGINT   DEFAULT NULL   COMMENT 'Redis索引内存占用(字节),通过FT.INFO获取',
    snapshot_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    KEY idx_time (snapshot_time)
) ENGINE=InnoDB COMMENT='向量索引快照';
```

### 3.4 检索权限过滤逻辑改造

**现状**：仅用 `@access_roles:{role}` TAG 过滤

**改造后**：综合考虑角色 + 岗位 + 部门 + 所属组织

```
Redis端过滤（FT.SEARCH）:
  (@access_roles:{ROLE_ADMIN}|@access_roles:{ROLE_HR})       -- 角色匹配（OR）
  (@access_positions:{3}|@access_positions:{5})              -- 岗位匹配（OR）
  (@access_depts:{10}|@access_depts:{15})                    -- 部门匹配（OR）

应用层后置过滤（Java端）:
  根据 access_mode 决定判定逻辑：
  - mode=0（任一满足）：角色/岗位/部门任一命中即通过
  - mode=1（全部满足）：角色 AND 岗位 AND 部门 全部命中
  - mode=2（角色+岗位同时满足）：角色 AND 岗位 必须同时命中，部门可选

组织过滤：SQL 和 Redis 中用 dept_id 过滤。
公司级过滤通过 dept_id 指向 parent_id=0 的部门节点来实现。
```

## 五、RAG 检索链路改造

### 5.1 检索流程（改造后）

```
用户提问
  │
  ▼
获取用户上下文: userId → roles[], positionId, deptId（用户所属部门，沿部门树向上找到 parent_id=0 的公司节点）
  │
  ▼
向量检索（Redis FT.SEARCH）:
  - KNN 向量相似度搜索
  - TAG 过滤: 角色 OR 岗位 OR 部门（根据 access_mode）
  - 组织过滤: dept_id 匹配
  - 只返回 vector_status=1 的文档
  │
  ▼
应用层二次过滤:
  - 校验 access_mode 逻辑
  - 校验版本号（version == MySQL version）
  │
  ▼
RAG Prompt 组装 → LLM 生成 → 返回
```

### 5.2 用户上下文获取

在 Gateway/JWT Filter 中解析 token 后，将用户完整上下文注入请求头：
- `X-User-Id`
- `X-User-Roles`（逗号分隔）
- `X-User-Position-Id`
- `X-User-Dept-Id`（用户所属部门ID，上层通过 `parent_id` 追溯到公司节点）

## 八、风险与注意事项

1. **索引重建耗时**：修改 Redis 索引 Schema 需要先删除旧索引再创建新索引，期间 KNN 检索降级到 MySQL，需在低峰期操作。
2. **嵌入 API 费用**：批量导入大量知识时会大量调用 SiliconFlow API，需关注配额和费用。
3. **权限复杂度**：角色+岗位+部门的三维权限叠加 access_mode 会让过滤逻辑变复杂，建议先实现 mode=0（任一满足），其余模式后续迭代。
4. **向后兼容**：`access_positions` 和 `access_depts` 默认 NULL 表示不限制，现有知识不受影响。
5. **Redis 单库限制**：Redis Stack 索引只能在 db=0 创建，如果后续有第二类向量（如审批单向量），需要共用 db=0 或用不同 prefix 区分。
