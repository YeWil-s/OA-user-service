## 二、项目建设目标

### 2.1 功能目标

完成一套覆盖企业日常办公场景的智能 OA 系统，至少实现以下功能：

- 用户登录、退出登录、当前用户信息查询。
- 用户、部门、岗位、角色、菜单、权限管理。
- 基于 RBAC 的菜单权限、按钮权限、接口权限控制。
- 员工管理、员工离职冻结、密码重置。
- 班次管理、上下班打卡、考勤记录查询、考勤统计。
- 请假、加班、外出等审批申请提交与审批处理。
- 公告发布、公告查阅、已读未读状态、站内消息中心。
- 员工档案、人事变动、固定资产登记、资产领用归还。
- AI 智能办公助手，支持智能填单、数据分析、OA 知识问答。
- 数据可视化大屏，展示人事、考勤、审批等统计数据。
- Web 管理后台与移动端快速办公能力。

### 2.2 技术目标

- 建立 Spring Cloud Alibaba 微服务工程骨架。
- 使用 Gateway 统一入口，完成统一路由、跨域和 Token 校验。
- 使用 Spring Security + JWT 完成无状态认证鉴权。
- 使用 MyBatis Plus 完成各业务数据库访问。
- 使用 Redis 完成缓存、分布式锁、Token 黑名单等能力。
- 使用 Nacos 完成服务注册发现和配置中心接入。
- 使用 OpenFeign 完成服务间调用。
- 使用 XXL-Job 完成考勤统计、数据聚合等定时任务。
- 使用 Vue3 构建可用、清晰、权限联动的后台管理系统。
- 使用 Docker Compose 提供本地一键启动方案。

## 四、总体架构任务

### 4.1 系统架构

```text
Vue3 管理后台 / Android 客户端
              |
      Spring Cloud Gateway
              |
  user-service        attendance-service
  approval-service    notice-service
  asset-service       ai-service
  visual-service
              |
 Spring Cloud Alibaba + Nacos + OpenFeign
              |
```

数据库设计要求：

- 所有表统一包含 `id`、`create_time`、`update_time`、`create_by`、`update_by`、`deleted` 等基础字段。
- 重要业务表应设置唯一约束，例如工号、角色编码、资产编号、同一员工同一天打卡记录。

- 敏感字段不保存明文密码，密码必须使用 BCrypt 加密。
- 所有跨服务统计数据通过定时聚合或接口调用同步，不直接破坏业务库边界。

## 八、安全与权限专项任务

| 编号 | 任务 | 说明 | 验收标准 |
| --- | --- | --- | --- |
| SEC-01 | Spring Security 配置 | 开启认证鉴权、白名单、跨域、无状态 Session | 未登录访问受保护接口返回 401 |
| SEC-02 | JWT 生成与校验 | Token 中包含 userId、username、roles、permissions、exp | 有效 Token 可访问接口，过期 Token 被拒绝 |
| SEC-03 | JwtAuthenticationFilter | 解析 Authorization Bearer Token 并写入上下文 | Controller 和权限注解可获取当前用户 |
| SEC-04 | RBAC 权限模型 | 用户、角色、菜单、权限多对多关系 | 不同角色看到不同菜单和按钮 |
| SEC-05 | 方法级权限 | 使用 `@PreAuthorize` 控制关键接口 | 无权限用户无法调用管理接口 |
| SEC-06 | 数据权限 | 支持全部、本部门及下级、本部门、本人 | 主管只能查看授权范围内的数据 |
| SEC-07 | 登录安全 | 失败次数限制、Token 黑名单、防重复登录可选 | 登录异常有明确错误提示并可审计 |
| SEC-08 | 基础防护 | 参数校验、SQL 注入防护、XSS 过滤、CORS 配置 | 常见非法输入不会破坏系统 |

### 10.2 核心验收流程

1. 管理员登录系统，创建部门、岗位、员工、角色和菜单权限。
2. 普通员工登录后只能看到个人相关菜单。

8. 用户输入自然语言申请，AI 返回结构化申请表单。
9. 数据大屏展示人事、考勤、审批统计数据。
10. 系统通过 Docker Compose 或本地启动脚本完成部署演示。

### 12.2 Commit 规范

```text

docs(readme): 更新项目启动说明

```

### 12.3 代码规范

- Controller 只处理参数接收和响应返回，业务逻辑放入 Service。
- Service 方法命名清晰，复杂逻辑拆分私有方法。
- Mapper 使用 MyBatis Plus，避免拼接 SQL。
- DTO、VO、Entity 分层明确，不直接把 Entity 暴露给前端。
- 所有新增接口必须补充参数校验和权限控制。
- 统一使用 `Result<T>` 返回数据。
- 异常统一抛出业务异常，由全局异常处理器转换响应。
- 敏感配置使用环境变量或配置中心，不提交 API Key、密码、Token。

## 十四、风险与应对措施

| 风险 | 表现 | 应对措施 |
| --- | --- | --- |
| 微服务拆分导致联调复杂 | 服务多、接口多、依赖多 | 先完成公共规范和接口文档，再模块化开发 |
| Spring Security 权限配置复杂 | 登录正常但接口 401/403 | 先打通最小登录链路，再逐步增加权限规则 |
| AI 接口依赖外部服务 | API Key、网络、费用限制 | 封装 AI Client，提供 Mock 模式或可替换模型配置 |
| 数据可视化缺少真实数据 | 大屏空数据或展示效果差 | 提供初始化演示数据和统计聚合任务 |
| 移动端工作量较大 | 影响核心系统完成度 | 以快速打卡、消息、申请审批为最小可用范围 |
| Docker 部署环境差异 | 本地能跑，容器不可用 | 第 6 周开始同步维护 Dockerfile 和 compose 文件 |
| 团队接口不一致 | 前后端联调效率低 | 统一响应格式、统一路由前缀、每个接口补充文档 |
